# ADR-0010. Приём ключей: конверт становится асимметричным — accepted 2026-07-10 (v2), уточняет 0004

Контекст (🔴 дыра, найдена трассировкой сценария А в MFC-000): по ADR-0004 master-key живёт
ТОЛЬКО у оркестратора — но ключ биржи в систему принимает core (одноразовая форма, S8).
Симметричным master-ключом core шифровать не может — его у core нет и быть не должно.
Слать plaintext оркестратору = дать оркестратору HTTP-вход и второй путь утечки.

Решение (предлагается): master-key — это КЛЮЦЕВАЯ ПАРА (age/NaCl sealed box, не самодельное).
- Публичная половина — в env core: на приёме ключа core генерит DEK, шифрует секрет AES-GCM,
  DEK запечатывает публичной половиной, пишет в key_ciphertext. Расшифровать обратно core
  НЕ может физически. Plaintext живёт миллисекунды в памяти обработчика запроса.
- Приватная половина — только в env оркестратора: распечатывает DEK в момент деплоя (как в 0004).
Ротация: новая пара → оркестратор перепаковывает DEK'и (ключи бирж не пересоздаются) —
механика 0004 сохранена. Аудит доступа — без изменений.

Довод: закрывает интейк без нового шва и без входящих портов у оркестратора; свойство
«core не умеет читать секреты» становится криптографическим фактом, а не дисциплиной.

Альтернативы: интейк через оркестратор (новый HTTP-вход у компонента с приватным ключом —
расширение поверхности атаки ровно там, где больнее всего); core держит симметричный
master-key (нарушает 0004: компрометация core = компрометация всех ключей).

Статус: уточнение ADR-0004 (формулировка «master-key» → «master-пара»); 0004 не отменяется.

## v2 — после адверсариального прогона (accepted 2026-07-10)
- **«Крипто-факт» ≠ гарантия безопасности (SEC1).** То, что core не может расшифровать ключ,
  верно буквально, но не защищает: скомпрометированный core (или operator-сессия) выбирает ОБРАЗ
  и цель deploy-job, а честный оркестратор расшифровывает ключ жертвы и инжектит его в контейнер
  атакующего. Компрометация core = утечка ключей, несмотря на асимметрию.
- **Ответ: привязать распечатку DEK к аттестации цели.** Расшифровка разрешена только для кортежа
  (account_id + client_id + allowlist-digest образа); bot_types.image — по реестру подписанных
  дайджестов, не произвольная строка; регистрация bot_type и назначение образа — привилегия с
  аудитом; развести право «создать job» и право «назначить образ».
- Severity для v1 — 🟡 (один доверенный Оператор, закрытый набор образов), 🔴-latent при 2-м
  операторе или динамической регистрации bot_type. Механизм allowlist закладываем сразу (дёшево в схеме).
- **Живой plaintext в env Railway (SEC2)** — отдельный сток, не решается 0010; см. threat-model п.1/п.6,
  очистка vars на teardown/ротации в ADR-0009v2. Формулу в flows/README «ключ живёт мс» — исправить (сделано).
