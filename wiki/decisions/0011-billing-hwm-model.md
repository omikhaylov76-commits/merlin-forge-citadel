# ADR-0011. Модель биллинга HWM — accepted 2026-07-10 · **финализирована 2026-07-14 (#27)**

Контекст: закон №8 требует тестов биллинга против модели. Направление принято 2026-07-10; формула
финализируется перед кодом Ф3. Оператор согласовал модель дословно (#23), Куратор ратифицировал
формулу и снял гейт «деньги» на билд движка (#27). Числа-примеры — эталонные тесты (обязательны).

## Модель (согласована Оператором, ратифицирована #27)

**Уровень:** расчётный период на `exchange_account`. equity = весь счёт в валюте договора (USDT/USDC).
**«Прибыль»** = ЧИСТЫЙ торговый заработок бота за период. Пополнения/выводы клиента — НЕ прибыль и НЕ убыток.
**HWM с переносом убытка:** комиссия только с новой прибыли ВЫШЕ прошлого пика; убыток периода переносится
и вычитается из будущей прибыли; комиссия — с остатка сверх пика. HWM вниз НЕ идёт (кроме сдвига на потоки).

### Каноническая форма (в пространстве накопленного профита) — так считает движок

Для периода t (границы UTC, непересекающиеся):
```
period_net_trading = end_equity − start_equity − (deposits_t − withdrawals_t)   # только торговый Δ
cum_profit_t       = cum_profit_{t−1} + period_net_trading                       # cum_profit_0 = 0
gross_over_hurdle  = max(0, cum_profit_t − hwm_{t−1} − hurdle_amount)            # hurdle=0 в v1
commission_t       = fee_pct × gross_over_hurdle                                 # fee_pct — СНАПШОТ договора
hwm_t              = max(hwm_{t−1}, cum_profit_t)                                # пик не опускается
```
mgmt-fee (%/год) — в модели, в v1 = 0 (заложить поле, не начислять).

### Потоки средств (депозит/вывод) — АБСОЛЮТНО (развилка 1, #24 принято)

Потоки исключены из `period_net_trading` → на `cum_profit` и HWM-в-профите НЕ влияют (долив ≠ прибыль,
вывод ≠ убыток). Это эквивалентно тому, как Куратор описал Оператору в equity-пространстве: планка HWM
сдвигается ровно на сумму потока (`hwm += deposit`, `hwm −= withdrawal`). Пропорционально к equity —
возможный v2 (ремарка), сейчас НЕ реализуем.

### Эталонные проверки (обязательные тесты, #27; hurdle=0, mgmt=0)

- start=10000; мес1 end=7000 (−3000) → commission=0, hwm держит (cum=−3000, пик 0);
  мес2 end=15000 (+8000) → cum=5000 → commission = fee_pct × 5000 (перенос убытка). ✓ (#27 сверил)
- cum доходит до 10000 → комиссия с 10000; далее +3000 → комиссия с 3000.
- депозит: планка вверх, не прибыль (комиссии нет). вывод-абсолют: планка вниз, не убыток.

## Прочие инварианты (ратифицированы #27)

- **fee_pct — СНАПШОТ** тарифа договора в период (не задним числом при смене тарифа).
- Закрытый период **immutable**; tz=UTC; периоды не пересекаются; корректировки (`adjustments`) типизированы.
- **Источник equity (MON3):** для РЕАЛЬНЫХ денег equity/закрытие — из АВТОРИТЕТНОГО источника (сверка с
  биржей), НЕ из телеметрии бота (её для денег не доверяем). v1/демо: засеянные demo-equity в тестах;
  сверка-с-биржей — поведение go-live.
- **Каденция (развилка 2):** v1 = календарный месяц (UTC) для всех. Колонку `billing_period`
  (enum month/quarter, default month) СОХРАНИТЬ в договоре — включение per-договор позже без миграции
  (#26). Логику quarter в v1 НЕ реализуем.
- **Модели оплаты:** profit_hwm (default) / capital_fixed / hybrid / subscription. v1 реализует profit_hwm;
  остальные — колонка есть, логика позже.

## Данные под статистику (#23-доп)

Период-леджер чистого торгового профита храним на **инстанс+клиент во времени** (не только текущее
HWM-состояние) → роллап сверху без переделки: «этот бот → клиенту» (per-instance), «все боты клиента»,
«всё по счёту/платформе» = суммы. Эффективность (профит/капитал) — производная, считается сумматором; экран — Ф4.

## Аудит (закон №4)

Расчёт комиссии — событие; закрытие периода → строка леджера + запись аудита («комиссия рассчитана» —
триггер будущих алертов). Отклонённые/ручные корректировки — тоже аудит.

Отклонено: доверять телеметрии бота как источником истины для денег; неттинг счетов клиента без согласия;
пропорциональный к equity сдвиг HWM на потоки (возможный v2). Тесты HWM пишутся ПОСЛЕ этой фиксации (закон №8).
Биллинг = деньги → независимое адверсариальное ревью ПЕРЕД merge обязательно.
