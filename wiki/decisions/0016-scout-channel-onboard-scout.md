# ADR-0016 — Scout-канал Контракта (v1): «Разведчик на борту, исправленный»

## Контекст
Ф5 «Разведка»: консоль показывает kanban сетапов (Forming/Tracking/Ready/Committed), но экран пока
**фикстура** (recon #49). Движок Пифагора при этом реально считает богатую разметку сетапа — точки
импульса A→B, Fib-ноги 0.382/0.5/0.618, стоп, `score`, `bars_since_anchor`, ТФ 4h/1h — а до платформы
она **не доезжает**: Контракт v0 несёт только торговую телеметрию (heartbeat/equity/trades/events,
ADR-0002), адаптер картриджа читает `build_monitor`, а не `build_scout`.

Куратор прогнал состязательную проверку двух путей (2 пути × атаки × синтез); Оператор ратифицировал
**«Разведчик на борту, исправленный»**: платформа рынок НЕ сканирует, только **принимает снимки**;
штатный vendored-скаут `scout/` включается ТРЕТЬИМ процессом картриджа (позже, #51+). Альтернатива
«свой сканер в платформе» **ОТВЕРГНУТА** — две копии стратегии = тихое расхождение показа и торговли,
обнаруживаемое уже после того как деньги сходили (№1/№6 разбора). Полный вердикт с ремонтами и рисками —
Project `citadel/scout-architecture-verdict-2026-07-16.md`.

## Решение

### (а) Закон №6 — уточняется, не отменяется
Активация штатного **keyless**-модуля `scout/` вендоренного снимка **ИЗ ОБЁРТКИ** (`start.sh`,
`bots/pifagor-cartridge`) — допустимая интеграция по Контракту, а не правка движка. Буква
«интеграция — только адаптером телеметрии» **УТОЧНЯЕТСЯ**: обёртка вправе поднять штатный процесс
движка и читать его опубликованное состояние; она по-прежнему **не импортирует и не меняет** код движка
(vendor неприкосновенен, ADR-0001). Закон №1 **чист** — в core приходит только JSON по контракту, не код
бота. Закон №2 **усиливается** — скаут keyless по построению (Bybit v5 public REST, без ключей;
`scout/public_api.py`), новый процесс не расширяет поверхность секретов.

### (б) Scout-канал = пятый телеметрический канал Контракта (v1)
К четырём каналам v0 (heartbeat/equity/trades/events) добавляется **ПЯТЫЙ — scout**: движко-нейтральный
СНИМОК сетапов. **CONTRACT_VERSION бампится v0→v1** (в обоих картриджах). Схема
`contracts/telemetry-scout.schema.json`: снимок **per (instance, symbol, tf)**, **replace-семантика**
(upsert — полный текущий набор заменяет прежний, НЕ append; ровно почему events не подошёл: канбан требует
замену состояния, а не журнал). Приёмная ручка ядра / таблица / readout — **#52** (ядро только записывает,
enforcement версии по-прежнему отложен).

### (в) Обязательные условия включения (fail-closed; без ЛЮБОГО — скаут НЕ включаем)
1. **Fail-closed гейт.** `start.sh` поднимает скаут ТОЛЬКО при явном `SCOUT_ENABLED=1` из env. Vendor-дефолт
   `True` (`scout/config.py:12`) НИКОГДА не решает за нас — обёртка передаёт значение явно.
2. **Отдельная `scout.db` (env `DB_PATH`).** SQLite `acquire_singleton` (`storage/db.py:365`) — flock на ОДИН
   файл, ключ игнорируется; общий файл с движком = лок-коллизия и смерть скаута через ~90с. Скаут пишет в свою БД.
3. **config_mismatch-детект.** `scan.py` зовёт детектор **литералами** (`ema=False, shorts=False` — `scan.py:79`;
   `reanchor=True` — `scan.py:28`; `STOP_FIB` из env — `scan.py:72`), а движок торгует по runtime-effective
   `ConfigStore`. Адаптер сравнивает эти 4 крутилки; снимок несёт `config_mismatch{flag,details}`; консоль
   **ОБЯЗАНА** показывать плашку при `flag=true` (показ мог разойтись с торговлей).
4. **Супервизор скаута в `start.sh`.** Liveness по `scout_control.heartbeat` + RSS-кап; рестарт ТОЛЬКО процесса
   скаута (`restartPolicy=never` на контейнере — OOM контейнера убил бы движок с позициями). Скаут — жертвенный,
   движок — защищён.
5. **Разведение бёрстов env'ами.** `SCOUT_RPS=1`, `SCOUT_CAL_UTC_HOUR` per-instance, умеренный `SCOUT_LIST_MAX`.
   Vendor-дефолты дают пики ~90 rps с общего IP — недопустимо для флота на одном egress.
6. **Режим «представителя».** `SCOUT_ENABLED=1` у ОДНОГО бота на группу идентичных профилей (drift=[]); молчащие
   боты шлют снимки со СВОИМИ `orders`/`position`, но БЕЗ `findings`; консоль подписывает «уровни от скаута X».

### (г) Гейт go-live
Скауты на ОБЩЕМ mainnet-IP боевых ботов опасны: **403-бан Bybit — IP-уровня**, ударит по управлению
позициями (не только по скауту). Поэтому на боевых ботах scout включается ТОЛЬКО отдельным решением Куратора,
до появления static egress / Docker-драйвера. Триггер пересмотра — как в ADR-0015.

### (д) Честная формула верности
Снимок faithful в смысле «**тот же код и данные того же сорта; конфиг совпадает на дефолтном профиле,
отклонение подсвечивается**» — НЕ «то же самое по построению». Известные ограничения снимка — НЕ скрытый
дефект, а **wish-list апстриму** `pifagor-v81`:
- литералы детектора (`scan.py`: `ema=False`/`shorts=False`/`reanchor=True`) — скаут не читает per-instance
  override этих крутилок; расхождение ловит `config_mismatch`.
- 4h-конвенция трекинга скаута vs 15m-интрабар исполнения движка — статусы (tracking/ready) могут расходиться
  **по существу** (скаут заливает по 4h-бару, у него нет 15m в Фазе 1).

## Механизм (этот шаг — ТОЛЬКО контракт + схема + тесты; 0 строк поведения)
- `contracts/telemetry-scout.schema.json` (v1): снимок-массив per (instance, symbol, tf); поля
  `symbol/tf/state/levels[{role,price}]/score/bars_since_anchor/klines[≤500]/klines_tf/orders/position/`
  `scan_ts/orders_ts/data_upto/detector_version/config_fingerprint/config_mismatch{flag,details}/producer`.
  Кап **≤500 свечей** (413 при превышении — ядро, #52).
- Весь Контракт помечен **v1** (`$id`/`title` всех схем), `CONTRACT_VERSION="v1"` в обоих картриджах
  (pifagor-cartridge + paper-bot). **Sync-гвозди** (тесты `tests/test_contract_version.py` в ОБОИХ картриджах):
  (1) версия-синк — `CONTRACT_VERSION` == версия из `$id` всех схем (не дают разойтись константе и схеме);
  (2) структурный гвоздь scout-снимка (pifagor) — множество полей/`required`/enum'ы схемы прибиты руками, дрейф
  структуры роняет тест. **⚠️ Отклонение-с-обоснованием (под подпись Куратора):** буквальное «schema ==
  **Pydantic**-зеркало» реализовано БЕЗ pydantic — полное Pydantic-зеркало scout-снимка живёт на стороне
  **ядра-приёмника (#52)**, где pydantic уже есть; картриджи намеренно без pydantic (снимок ПРОИЗВОДЯТ и
  валидируют jsonschema; `telemetry-schemas.md`), а ядро в этом шаге не трогаем. Структурный гвоздь (2) даёт
  реальный schema↔ожидаемая-структура гейт уже сейчас, в рамках ограничений. enforcement версии в ядре отложен.
- **0 строк в `vendor/`.** `start.sh` / adapter / ядро / консоль НЕ трогаются — это **#51** (изоляция
  картриджа), **#52** (ядро+readout), **#53** (консоль), по одной после приёмки каждой.

## Последствия
- Контракт растёт до **5 каналов**; ядро v1 принимает scout как запись (без enforcement версии).
- Включение скаута — за **6 условиями (в)**; любое невыполненное = не включаем (fail-closed).
- go-live скаута на боевом IP — **отдельный гейт (г)**.
- Расхождения показ/торговля **не скрыты** — подсвечены (`config_mismatch`) либо задокументированы как
  upstream wish-list (д).

## Триггер пересмотра
Как ADR-0015: Railway Pro / Docker-драйвер (static egress) / go-live (реальные ключи-деньги) → пересмотр
гейта (г) и разведения бёрстов (в.5). Плюс: правка литералов детектора в апстриме `pifagor-v81` → пересмотр (д).
