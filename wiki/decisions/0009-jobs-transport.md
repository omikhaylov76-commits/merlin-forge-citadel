# ADR-0009. Транспорт jobs: HTTP long-poll ядра — accepted 2026-07-10 (v2)

Контекст: architecture.md говорил «оркестратор слушает таблицу jobs» — неоднозначно: читать
таблицу напрямую или через API. Закон №3 допускает ОБА механизма (HTTP API и таблицы-контракты),
так что это честная развилка, а не вопрос законности.

Решение (предлагается): jobs идут через internal API ядра, как у ботов:
`GET /v1/internal/jobs/next?wait=25s` (long-poll, lease с lease_expires_at) → работа →
`POST /v1/internal/jobs/{id}/ack {done|failed, detail}`. Токен-скоуп `orchestrator`.
Идемпотентность — на ядре (unique-index «один активный deploy на инстанс», attempts, ре-lease).

Довод: (1) единообразие — бот и оркестратор суть одинаковые клиенты ядра, один паттерн
long-poll+ack, одна модель токенов (ADR-0008); (2) БД платформы остаётся с единственным
писателем — миграции не координируются между сервисами; (3) аудит доступа к конвертам ключей
пишет ядро в одном месте; (4) оркестратор тестируется мок-HTTP без поднятия Postgres.

Альтернатива — прямое чтение таблицы (SELECT … FOR UPDATE SKIP LOCKED): транзакционная
аренда «бесплатно», нет HTTP-звена. Отклоняется для v1: второй сервис прирастает к схеме
(любая миграция jobs = синхронный релиз двух сервисов), аудит расщепляется на двух писателей,
GRANT-хозяйство. На нашем масштабе выгода SKIP LOCKED не окупает сцепку.

Пересмотр: если internal API станет узким горлом (не при 10 инстансах) — SKIP LOCKED вернуть
как таблицу-контракт с выделенным ADR о совладении схемой jobs.

## v2 — после адверсариального прогона (accepted 2026-07-10)
- **Long-poll не держит БД-коннект (SCL1, 🔴).** Инвариант: ожидание — цикл проверка-сон с возвратом
  коннекта (или in-memory wakeup), НЕ Depends(get_db) на весь запрос. Нагруз-тест-гвоздь Ф1:
  «100 парковок + stop_close проходит <1с». Иначе флот вешает пул и убивает свой же kill-switch.
- **Fencing + верификация деплоя (OPS2, 🔴).** ack принимает ТОЛЬКО держатель актуального lease
  (fencing-токен). После serviceCreate — перечитать сервисы по имени, дубль удалить (create не
  атомарен с поиском; уникальность имени Railway не гарантирована — проверить в Ф1).
- **teardown не терминален при живом сервисе (OPS5, 🔴).** destroy идемпотентен (404=успех);
  teardown-job не уходит в failed, пока `status(infra_ref)` показывает живой сервис — бесконечный
  backoff + эскалация. «3 фейла→failed» остаётся только для deploy.
- **≤1 живой инстанс на счёт (OPS3, 🔴).** Partial unique index в домене; failed_deploy ОБЯЗАН
  породить teardown частичных ресурсов (сервис+схема+роль); heartbeat к failed_deploy → «зомби ожил».
- **restartPolicy=never + sticky stop_close (OPS1, 🔴).** Бот-сервис не перезапускается инфрой;
  пока instance в stopping — /commands/next отдаёт stop_close. Гасить сразу по ack=ok, не через задержку.
- **Прочее:** отпуск lease по недоступности инфры ≠ attempts++ (OPS16); 5-мин таймер старта — от
  serviceCreate, не от постановки job (SCL6); параллелизм оркестратора — крутилка (SCL6);
  усыновление сироты = перезапись ВСЕХ vars (свежий токен) + redeploy, не голый infra_ref (OPS12).
