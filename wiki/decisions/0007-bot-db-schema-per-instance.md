# ADR-0007. БД ботов: отдельный кластер, схема-на-инстанс — accepted 2026-07-10 (v2)

Контекст: движкам (Пифагор) нужен свой Postgres для внутреннего состояния. Допущение №4
handoff'а S1 («схема-на-инстанс в одном Postgres») требовало решения. БД ботов ≠ БД платформы.

Решение (предлагается): ДВА разных Postgres. №1 — платформенный, владеет только core (S6).
№2 — «кластер ботов»: одна БД, на каждый инстанс оркестратор при деплое создаёт схему
`inst_{id}` + роль с правами только на неё (search_path зажат). Движок сам мигрирует свою
схему при старте (Контракт). При teardown роль отзывается сразу, схема живёт N дней для
разбора, потом чистится. Admin-URL кластера ботов — только в env оркестратора.

Довод: физическая изоляция ботов от данных платформы (угроза №4 закрывается не токенами,
а отсутствием пути); один кластер = один бэкап и ~$10/мес вместо инстанса-на-бота;
изоляция ботов друг от друга — ролями (достаточно для наших же движков за Контрактом).

Альтернативы: БД-на-инстанс в том же кластере (чуть жёстче изоляция, больше возни с
провижионингом/бэкапом — не даёт нового качества на 10 инстансах); Postgres-сервис Railway
на каждого бота (кратный рост цены, зоопарк бэкапов); SQLite в контейнере (теряем состояние
при пересоздании сервиса — для живых позиций недопустимо).

Пересмотр: при недоверенных сторонних движках — вернуться к БД-на-инстанс.

## v2 — после адверсариального прогона (accepted 2026-07-10)
- **Убрать порог «>30 инстансов» (SCL2).** Он противоречил vision «×10 (~100) без перестройки».
  Схема-на-инстанс с пулером держит 100 спокойно; триггер пересмотра — только недоверенные движки.
- **Потолок коннектов (SCL2).** 100 движков × пул = 200–500 подключений к одному Postgres (потолок
  ~100). Ответ: Контракт Бота требует pool_size ≤ 2; pgbouncer (transaction mode) — заглушка-труба
  перед кластером ботов, включается по росту.
- **Жёсткая изоляция роли (SEC8/OPS10).** Провижининг: `REVOKE ALL … FROM PUBLIC`, выдать
  USAGE/CREATE только на `inst_{id}`, `search_path=inst_{id}`, per-role connection limit; тест-гвоздь
  «роль A не видит объектов схемы B». Мониторинг размера схем/диска кластера с алертом по порогу
  (один сбойный движок не должен переполнить диск всем — коррелированный отказ).
