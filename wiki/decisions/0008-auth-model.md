# ADR-0008. Auth-модель API ядра — accepted 2026-07-10 (действует v2)

> ⚠️ Принят в редакции **v2** (см. низ файла): ЕДИНЫЙ механизм opaque-токенов для всех
> принципалов. Блок «Люди: JWT…» ниже — исходное предложение, ЗАМЕНЕНО v2, не реализовывать.

Контекст: у одного API пять принципалов: Оператор (консоль), клиент (портал), бот-инстанс,
оркестратор, дирижёр-ансамбля (труба). Нужна модель, где скоупы узкие, отзыв мгновенный,
а сложность — по масштабу (10 клиентов, YAGNI).

Решение (предлагается): люди и машины аутентифицируются РАЗНО.
- Машины (бот, оркестратор, дирижёр): opaque-токены — случайные 256 бит, в БД только hash.
  Скоуп в строке токена: `instance:{id}` / `ensemble:{id}` (труба, v1 отвергается) /
  `orchestrator`. Отзыв = удаление строки (мгновенно, при stop/ротации). Без TTL, ротация
  при передеплое инстанса.
- Люди: JWT access 15 мин + refresh с ротацией; Оператор — обязательный TOTP (угроза №2),
  клиенты v1 — пароль (выдаёт Оператор, managed-модель, самосервиса нет). Rate-limit логина,
  алерт на вход Оператора с нового устройства, каждое auth-событие — в audit_log.
- RBAC: роль в claim'е; портальные ручки дополнительно проверяют владение строкой
  (client_id токена = client_id ресурса) — двухслойно, роль сама по себе не пускает к чужому.

Довод: opaque для машин бьёт JWT по главному критерию — мгновенный отзыв без денылистов;
JWT для людей избавляет от серверных сессий при двух SPA на разных origin'ах.

Альтернативы: JWT для всех (отзыв бот-токена требует денылист — та же БД, больше движущихся
частей); серверные сессии-cookie (два origin'а, CORS-грабли); mTLS для машин (котлован).

## v2 — после адверсариального прогона (accepted 2026-07-10, действующая редакция)
- **★ Свернуть двойной стек в ОДИН механизм (SCL7).** JWT+refresh+ротация для 1 Оператора и ~10
  клиентов — переинженерия, и заявленный «мгновенный отзыв людям» им НЕ достигается (JWT живёт до
  15 мин). Решение: opaque-токены в БД (hash) + скользящий TTL для ВСЕХ принципалов (люди и машины);
  TOTP Оператора сохраняется. Отзыв мгновенен и единообразен; минус JWT-библиотека, refresh-эндпоинт
  и класс гонок ротации. Портальные ручки всё равно бьют в БД за проверкой владения — lookup не лишний.
- **Проверка владения — на ВСЕХ ручках, не только портал (SEC7).** token.subject == ресурс.owner;
  cmd_id/все id = UUID (закрыть перебор); ack чужого cmd → 403 + audit + алерт.
- **Двухфазная ротация токена (OPS6).** Новый токен — вторая активная строка; старый отзывается
  только после первого heartbeat нового контейнера. Иначе живой торгующий бот теряет PAUSE/STOP.
- **stop_close — настоящий 2-й фактор клиента (SEC6, ↓🟡).** TOTP/e-mail-код/подтверждение Оператора,
  не повтор POST той же сессии. Иначе фишинг пароля = рыночное закрытие позиций клиента.
