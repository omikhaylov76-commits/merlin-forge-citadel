# ADR-0018 — Настройки дозора из ядра (генная фича)

## Контекст
«Разведка-стол» (S7): Оператор крутит пороги дозора (скаута) в панели консоли; их надо durable-доставить
картриджу-представителю (Галахад), применить и мягко перезапустить ТОЛЬКО скаут. Два факта разведки:
scout.db эфемерна; у Галахада нет durable-тома (том — только для торгового состояния боевых, П1); а
эндпоинт настроек операторский (картридж по instance-токену его не читает).

## Решение (одобрено Куратором)
**Ядро = вечная истина, картридж = stateless-применитель; настройкам том НЕ нужен.**
- Ядро: таблица `scout_settings` (desired) + журнал (audit_log) + **instance-scoped** `GET /v1/scout/
  settings/self` (по instance-токену, паттерн `/commands/next`).
- Картридж: на **boot** (фоновый поток) и на команду `dozor_apply` собирает shell-sourceable файл
  `SCOUT_*`-env + бампает gen-файл; супервизор `scout_supervise` source'ит файл на КАЖДОМ (ре)старте
  скаута и по смене gen мягко рестартит ТОЛЬКО скаут (движок/адаптер целы). `scan_now` — вендорский
  `scout_control_mark(scan_now_ms=…)`.
- **Закон эталона (ADR-0014):** механизм — в исходнике картриджа (start.sh + app), значит КАЖДЫЙ бот от
  Мерлина рождается с умением «забрать свои настройки из ядра». Генная фича.

## Три стража (обязательны, в коде — не по договорённости)
1. **Boot-fetch не блокирует.** Скаут стартует на генных дефолтах start.sh; fetch ретраится фоном, по
   успеху gen-рестарт. Движок от этого канала не зависит вообще (guard: поток daemon, only if scout on).
2. **Анти-инъекция.** Sourceable-файл собирается ТОЛЬКО из whitelist SCOUT_*-ключей; значения строго
   приводятся к int/float/enum (`to_env_lines`), сырая строка из сети в файл НЕ попадает (тест на
   `"180; rm -rf /"` → пропуск). Закрывает классическую дыру «source чужого текста».
3. **Скоуп канала.** Через self-readout ходят ТОЛЬКО SCOUT_*-пороги дозора. Движок/риск (LIVE_TRADING,
   COINS_CONFIG, риск-пороги) этим каналом не ходят НИКОГДА — их путь — профили Конструктора (отдельная
   веха, отдельные гейты). Whitelist зашит в `_INT/_FLOAT/_TF` картриджа.

## Часовой автоскан (Q1, проверено в вендоре)
`SCOUT_TF` (primary) гонит границу авто-скана (`_cur_boundary`, main.py). `SCOUT_TF=1h` → часовые
границы → Этап B каждый час сканит ОБА ТФ (`scanned_tfs`) → 4h-сетапы обновляются чаще, курсор пуша #54
держится. start.sh прокидывает `SCOUT_TF` (раньше не прокидывал) — аддитивно, дефолт `4h`.

## Последствия
- +1 аддитивный instance-эндпоинт, +1 таблица (0013), +картриджные start.sh/app (генно). 0 vendor.
- Боевой контур не задет: движок/торговля независимы; рестартится только скаут-процесс.
- Расхождение ядро↔картридж (команда не доехала) видно по статусу `dozor_apply` в строке дозора.
- НАБОР-2/движок этим каналом не управляются (страж 3) — мост в бота остаётся отдельной вехой.
