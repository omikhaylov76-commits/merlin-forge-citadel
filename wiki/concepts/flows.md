# Сквозные потоки (2 контрольных сценария)

Номера швов — из [seams](seams.md). Дыры, найденные трассировкой, помечены 🔴/🟡 и сведены
в отчёт MFC-000 (⛔ Куратору).

## А. «Оператор разворачивает инстанс Пифагора клиенту X»

1. **[S8]** Клиент по одноразовой ссылке вводит API-ключ → core шифрует конвертом (публичной
   частью master-ключа, ADR-0010), plaintext умирает с запросом; `exchange_accounts` + аудит.
   🔴 Дыра-первоисточник: по старой букве ADR-0004 core не мог шифровать вовсе → ADR-0010.
2. **[S1]** Оператор в консоли: клиент → счёт (права ключа проверены, `perms_checked_at`) →
   выбор паспортизированного профиля → `POST /v1/instances` → ядро: instance(status=pending),
   job(deploy), аудит. Консоль НЕ знает про Railway.
3. **[S3]** Оркестратор арендует job → тянет конверт ключа + профиль через internal API →
   аудит «доступ к ключу» (ADR-0004).
4. **[S7]** Оркестратор провижионит схему+роль в БД ботов → BOT_DB_URL.
5. **[S5]** `driver.deploy(image=allowlist-digest, env{MF_*, EXCHANGE_*, BOT_DB_URL}, name=mfc-inst-{id})`
   → infra_ref в instance; статус=starting. ⚠️ Расшифрованный ключ живёт в памяти оркестратора
   на шаге 4–5 И далее ПЕРСИСТЕНТНО в переменных сервиса Railway весь срок инстанса (SEC2 —
   не «миллисекунды»; сток plaintext, см. threat-model п.6/п.9). Очистка vars — на teardown/ротации.
6. **[S4]** Контейнер стартует: движок мигрирует свою схему, шлёт 1-й heartbeat → ядро
   переводит starting→running. Консоль видит зелёный инстанс через read-model.
7. Отказ на любом шаге — таблица отказов в seams; ключевое: не-старт за 5 мин = failed_deploy.

Находки: 🟡 лимит размера env на Railway для MF_PROFILE_JSON — проверить на paper-bot (Ф1);
🟡 узкий GRANT оркестратора в internal API (конверты + jobs, не весь домен).

## Б. «Клиент жмёт STOP_CLOSE»

1. **[S2]** `POST /v1/portal/instances/{id}/stop_close` → ядро проверяет владение → выдаёт
   challenge_id (серверное двойное подтверждение) → повторный POST ≤60с. Обе стадии — аудит;
   Оператору мгновенный алерт (outbox→Telegram, ADR-0005).
2. Ядро: command(stop_close, cmd_id) в очередь инстанса; instance.status=stopping (намерение).
3. **[S4]** Бот забирает команду long-poll'ом → закрывает позиции безопасным путём движка →
   финальный flush телеметрии → `ack(result)` → выходит.
4. **[S3]** Ядро на ack **с result=ok** создаёт job(teardown) → оркестратор гасит сервис (S5),
   через N дней чистит схему бота (данные держим для разбора); status=stopped. Портал показывает
   стадии честно: «команда отправлена → принята ботом → позиции закрыты».
   ⚠️ result=error («закрыть не смог», OPS4) → status=stopping_failed, БЕЗ teardown, алерт + runbook
   «закрыть руками»; портал: «закрыть не удалось, идёт ручное закрытие». Гасить с открытой позицией нельзя.
5. 🔴 Бот мёртв и команду не заберёт: платформа НЕ умеет закрыть позиции сама (у core нет
   выхода на биржу — by design). Таймаут ack 10 мин → алерт + runbook «закрыть руками»;
   финальный рубильник клиента — отзыв ключа на бирже (довод ADR-0005). Это ДОЛЖНО быть
   в тексте портала и в дисклеймерах (legal), иначе обещаем то, чего не гарантируем.

## Машина состояний инстанса (владелец переходов — core)

    pending → deploying → starting → running ⇄ paused
       ↘ failed_deploy ↙        running/paused → stopping → stopped
                                            ↘ failed

- **status** = жизненный цикл/намерение. Владельцы переходов: core ставит pending, running,
  paused (на ack pause), stopping, failed (рантайм-отказ живого инстанса), stopping_failed;
  оркестратор через internal API — deploying, starting, stopped, failed_deploy. Каждый статус
  имеет владельца (COH12). stopping без ack за таймаут → оператор может force-stop (перевод в
  stopped с причиной в audit, OPS17). Переход в любой статус — строка audit_log.
- **health** = отдельное поле, производное от свежести heartbeat (ok|stale|dead).
  Разделение убирает мигание статуса при сетевых икотах: «running + stale» ≠ «stopped».
- Любой переход = строка audit_log (закон №4).

## Биллинг-шов (обнаружен трассировкой, сценарий В — эскиз)

Закрытие billing_period читает equity из телеметрии → а её шлёт бот. 🔴 Потеря/враньё
телеметрии искажает HWM-комиссию. v1: закрытие периода — операторская операция со сверкой
по бирже (число с биржи вводится руками, расхождение > порога — блокирует закрытие).
Автосверку — в Icebox. Тесты HWM-математики обязательны (закон №8) и не зависят от этой дыры.
