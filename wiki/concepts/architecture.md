# Архитектура — обзор

Четыре модуля вокруг одной БД, боты — за Контрактом:

    ┌─────────┐   ┌────────────┐   ┌──────────┐
    │ console │──▶│            │◀──│  portal  │      боты (контейнеры)
    └─────────┘   │  core API  │   └──────────┘      ┌────┐ ┌────┐ ┌────┐
                  │  Postgres  │◀────телеметрия──────│ #1 │ │ #2 │ │ …  │
    ┌───────────┐ │  audit     │─────команды────────▶└────┘ └────┘ └────┘
    │orchestr.  │◀▶            │
    └─────┬─────┘ └────────────┘
          └─ драйвер: Railway API (v1) / Docker (позже)

Потоки: консоль и портал ходят ТОЛЬКО в core API. Оркестратор слушает задания ядра
(таблица jobs) и управляет контейнерами. Боты шлют телеметрию в core (push, токен инстанса)
и получают команды оттуда же (poll/long-poll — без входящих портов у ботов).

Домен (ядро): clients · exchange_accounts (ключи шифрованы) · bot_types · profiles(+паспорта)
· instances (node_id, ensemble_id — трубы) · telemetry (heartbeat, equity, positions, trades, events)
· billing_periods (HWM) · research_artifacts · jobs (деплой/бэктест) · audit_log · users/roles.

Трубы, лежащие в фундаменте с v1: драйвер оркестрации; поле exchange у счёта (+таблица
возможностей бирж); ensemble_id + командный API (дирижёр = просто ещё один тип бота с токеном
группы); outbox-события; node_id; RBAC; jobs-очередь (бэктесты из консоли позже).
