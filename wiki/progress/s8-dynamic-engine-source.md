## Диагноз (доказан живьём, 2026-07-22)

## Разведка (сверено построчно)

**Что уже есть в вендоре (читаем reader-паттерном, НЕ меняем):**
- `scout.main.decide`: Этап A (пересборка scout_list) — раз в сутки (5:00) / bootstrap; Этап B —
  **на КАЖДОЙ границе SCOUT_TF (4h) + кнопка**.
- `scout.scan.scan_list` (Этап B): **докачивает свежие свечи ВСЕХ монет scan_list** (~155 у Борса)
  в кэш `scout_klines` (rolling, `klines_cache.top_up`, ретеншн SCOUT_CAL_BARS=1000) на каждом
  скане. ⇒ **свежие свечи широкого пула УЖЕ в scout.db каждые 4h** — новой сети НЕ нужно.
- `storage.db`: `scout_list_all()` (курированный список: symbol+score+mb) и `scout_universe_all()`
  (метрики+rejects всей вселенной) — готовые читалки.
- `strategy.warm.classify` — движковый placeable-вердикт; адаптер уже зовёт его per-coin
  (`scout_reader._classify`/`_truth`, единая Разведка) — **переиспользуем, не плодим**.
- Фильтры КАЧЕСТВА Оператора (возраст≥90д/оборот/спред/история) уже применены вендором при
  сборке scan_list (Этап A `hard_rejects`) — пул «солидных» монет сохраняется, возраст не трогаем
  (решение Оператора).

**Ритмы совпадают:** Этап B (свежие свечи + наш пересчёт) и самоход движка (`WARM_EACH_CYCLE`)
живут на одной 4h-границе → placeable-монета попадает в coins.json к тому же тику, где самоход
её подхватит. Кнопка «Сканировать сейчас» ускоряет всё разом (скан→отбор→стек→постановка).

## Архитектура фикса (целиком 0-vendor, только НАШ адаптер)

**Источник широкой вселенной** = `scout_list` (155 монет качества, фильтры Оператора в силе).
**Критерий отбора в стек** = `warm.classify` по свечам кэша: PENDING (auto_eligible ИЛИ
reanchored) = placeable → кандидат. Приоритет при капе: auto → reanchored, внутри — по скору
списка. **Скаут = радар/витрина** (телеметрия Разведки не меняется), торговую вселенную больше
НЕ диктует.

Переключатель: `DYNAMIC_SOURCE` env, **дефолт `scout` = прежнее поведение байт-в-байт**
(fail-closed, как WARM_EACH_CYCLE); Борсу выставляем `engine`. Флот/Персиваль/Галахад не задеты
(у них DYNAMIC_ENABLED выкл вовсе).

## Миграция и откат
- Переключение = env на сервисе Борса + redeploy. Стек пересоберётся: не-placeable выйдут по
  exit-гистерезису (естественное убытие), held (открытые 1INCH/BANK/VVV) ЗАПИНЕНЫ — позиции
  не осиротеют (урок 18.07; positions-flag откладывает рестарт движка при живых ордерах — уже
  видели вживую «рестарт ОТЛОЖЕН»).
- Откат = `DYNAMIC_SOURCE=scout` (или снять env) + redeploy — прежнее поведение, код один.

## Аудит 5 линз (по заданию Куратора)
1. **Архитектура:** развязка = смена ИСТОЧНИКА кандидатов внутри существующего провайдера;
   разъём ADR-0019 (coins.json+gen+рестарт) нетронут, второй механизм не плодится. Канал
   ADR-0020 сохраняет поля, семантика двух — уточнена (дока). Скаут остаётся радаром — витрина
   Разведки уже построена на этом разделении (слои «радар vs движок» совпали с фиксом).
2. **Швы:** Контракт/ядро/консоль НЕ меняются (0 новых команд/полей/миграций). Меняется только
   адаптер (наш код). Телеметрия та же; консоль уже показывает placeable-вердикты.
3. **Баги/отказы:** scan_list пуст (до первого Этапа A) → placeable-скан пуст → ПОЛ-НА-ПУСТОТУ
   держит прежний набор (не пишем пустоту). Кэш свечей короткий по монете → _classify (False,
   None) → монета просто не кандидат (fail-soft). Ошибка одной монеты не роняет проход (как
   _truth). Мигание reanchor↔auto гасится гистерезисом входа.
4. **Флот не задет:** Персиваль/Галахад — DYNAMIC_ENABLED выкл (провайдера нет вовсе), их образ
   не передеплоим; дефолт `scout` = старый путь байт-в-байт (регресс-тест); геном 65-frozen
   неизменен, страж-дрейфа = 2 дельты (правок вендора НЕТ).
5. **Законы/секреты:** Закон 1/6 — 0-vendor (только вызовы); Закон 3 — модульность не тронута;
   Закон 5 — портал не видит (ничего клиентского); секретов нет (env-флаг — не секрет). Демо.
   **Открытые позиции:** пин held + positions-flag + отложенный рестарт — тройная защита, не
   меняется (линза Куратора закрыта).

## Открытый вопрос (не блокер плана)
Почему AKE/ENA/KAITO/USELESS/SIREN/UAI не в НАХОДКАХ скаута Борса (мой трейс с дефолт-mb давал
tracking, живой скаут с КАЛИБРОВАННЫМИ mb молчит / или монеты вне scan_list по скор-капу) —
проверится само на живой сверке: в engine-режиме находки скаута больше не гейтят стек, отбор
идёт напрямую classify по scan_list. Если монета не в scan_list — увидим в живой сверке и
решим отдельно (это уже про состав пула, не про критерий).
