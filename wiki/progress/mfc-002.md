# MFC-002 — core-scheduler «часовой»

**Веха:** Ф1 · **Ветка:** task/mfc-002 · **Последний коммит:** f698fd5

## Цель
Один asyncio-цикл-«часовой» в ядре + реестр свёрток (плагин-точка для периодических задач).
Первая реализованная функция — **dead-man тик**: цикл на каждом обороте штампует монотонное
время; `/healthz` показывает свежесть штампа (SCL3, seams.md:78–79). Умер цикл, а процесс жив —
это видно в /healthz. Порог «мёртв» = 3× периода тика (паттерн 3×60с из seams.md:62).

## Границы (чего в MFC-002 сознательно НЕТ)
- НЕТ таблицы instances / heartbeat и НЕТ реального stale-скана (нет схемы в 0001;
  `last_heartbeat_at` — это MFC-001-доп). Stale-скан — следующая свёртка, отдельная фича.
- НЕТ авто-рестарта ботов (seams.md:62 — позиции неизвестны).
- НЕТ нового шва наружу: часовой живёт внутри процесса ядра.
- Реестр свёрток shipped пустым, но доказан тестом (плагин-точка готова к stale-скану).

## Развилка на ⛔ (решает Куратор) — семантика /healthz при мёртвом цикле
- **(A, рекомендую)** ядро отдаёт 200, в теле `scheduler.state="dead"` + возраст тика.
  Алерт — через SCL3 outbox (позже), не рестартом. Минимально, обратимо, liveness не завязан
  на внутренний компонент. Зафиксировать как **ADR-0012**.
- **(B)** `/healthz → 503` при мёртвом цикле → Railway перезапускает процесс ядра (классический
  dead-man auto-recovery; ядро без позиций — рестарт безопасен). Связывает liveness с часовым.
