# S8 Борс — горн + самоход

**Статус: разведка ✅ → ПЛАН на подпись Куратора. Код НЕ начат (гейт: подпись; если геном — подпись ADR-0021).**

Задача Куратора (QUEUE 2026-07-21, «ОСНОВНАЯ ЗАДАЧА»): довести Борса-динамика до рабочего состояния —
сам находит свежие сетапы, проверяет годность (`auto_eligible`) и заходит. Без большой перестройки.
Витрина «правда движка» / Postgres-факт-флот / F-warm-button — ОТЛОЖЕНЫ (отдельная веха).

## Цель (спека = критерий приёмки)
Снять два тормоза (оба вскрыты мной в F-lookahead):
1. **Горн** — Оператор жмёт кнопку → Борс СЕЙЧАС прогоняет `warm_start_auto` по текущей вселенной →
   ставит `auto_eligible` в ордера. Не ждать автоподхвата.
2. **Самоход** — `warm_start_auto` на КАЖДОЙ границе SIGNAL_TF, а не 1 раз при старте (`main.py:184`).

**Жёстко (Оператор):** ритм из `SIGNAL_TF` (не хардкод «4 часа»). **Фильтр входа уже принят** (`c03fc2f`):
SIGNAL_TF-only + стадии tracking/ready + `auto_eligible` + кап-стек 10. Ставим ТОЛЬКО `auto_eligible`
(нетронутый PENDING); пере-якорь (F-warm-button) — НЕ здесь. Тейк-профит 1-й ноги — техдолг, НЕ здесь.
**Граница:** ведение уже открытых позиций не ломать (урок 18.07); Персиваль/Галахад/эталон байт-в-байт; демо.

## Разведка (факты из кода)
- **3 процесса** (`start.sh`): движок `vendor/app/main.py`, адаптер `картридж/app/main.py` (foreground),
  скаут. Общение — SQLite воркера + durable-интенты в `config_log`.
- **Команда:** ядро → адаптер `bot.py:_handle_command` (`pause`/`scan_now`/`screener_run`) → интент в БД →
  движок читает каждый тик. Адаптер логику движка НЕ зовёт (разные процессы) — только интент.
- **warm (движок):** `warm_start_auto` (`cycle.py:246`) ставит ТОЛЬКО `auto_eligible` (`_warm_one:277`
  проверяет `auto_eligible`) — зовётся 1 раз (`main.py:184`, гейт `WARM_ON_START`). `maybe_warm`
  (`cycle.py:319`) — каждый тик, консюмер интента `WARM_APPLY`, ставит через `_warm_one_button` = ЛЮБОЙ
  PENDING (пере-якорь тоже) = «кнопка человека» = **отложенный F-warm-button**, НЕ «авто-годен».
- **Петля движка** (`main.py:619-644`): `_poll_tick` (каждый тик, 15m) + `_maybe_4h_cycle` (на границе).
  `_maybe_4h_cycle` (`main.py:588-600`): `b=(now//FOUR_HOUR_MS)*FOUR_HOUR_MS`; на НОВОЙ границе →
  `run_4h_cycle` + `_write_warm_candidates`. **Граница — хардкод `FOUR_HOUR_MS`** (`klines_4h.py:24`,
  константа), НЕ из SIGNAL_TF. Готового `tf→ms` нет (`interval_ms()` = EXEC_INTERVAL 15m, без аргумента) →
  «ритм из настроек» требует крошечного парсера `SIGNAL_TF("4h"/"1h")→ms`.
- **Постановка ордера** (нужен живой `executor`) — ТОЛЬКО внутри движка. Адаптер влияет лишь интентами.

## 🔑 РАЗВИЛКА (на твою подпись): 0-vendor vs геном

| | **Б. 0-vendor** (адаптер) | **А. Геном** (Закон 6) — рекоменд. |

| Как | адаптер сам считает `auto_eligible` (`warm.classify` на свечах scout.db, как `_verified_grid`) → пишет `WARM_APPLY` по годным: на границе SIGNAL_TF (самоход, таймер адаптера) / по `scan_now` (горн). Движок ставит существующим `maybe_warm`. | хирургический хук в движке: авто-warm (`auto_eligible`) на границе SIGNAL_TF + по новому интенту (горн); граница из `SIGNAL_TF`, не `FOUR_HOUR_MS`. |
| ТФ-ритм | адаптером (0 генома) | из настроек движка |
| ADR | не нужен | ADR-0021 + страж-дрейфа + тесты + подпись |
| Минусы | (1) `_warm_one_button` НЕ проверяет `auto_eligible` → монета, ставшая пере-якоренной между чтением адаптера и движка, ВСЁ РАВНО ставится — **протёк отложенного F-warm-button**; (2) свечи адаптера (scout.db) ≠ свечи движка → **прокси** (для ПОСТАНОВКИ ордеров faithfulness слабее, чем для витрины); (3) «кнопочный» канал не по назначению. | правка генома → процесс Закона 6 (ADR + страж + тесты + подпись). |

**Рекомендация Инженера — А (геном).** Точная семантика `auto_eligible` (не протекает в отложенный
F-warm-button), реальные свечи движка, честная постановка. Правка хирургическая (несколько строк, 2 места
движка + 1 адаптер). 0-vendor соблазнителен скоростью, но для ПОСТАНОВКИ демо-ордеров прокси-свечи и
семантический протёк — риск, а не мелочь.

## План А (рекомендуемый) — ГЕНОМ
Последний коммит: 5e6800d

**Точки врезки:**
- **Самоход** — `vendor/app/main.py:_maybe_4h_cycle` (588-600): (а) границу считать от `SIGNAL_TF` (новый
  парсер `_signal_tf_ms()`), не `FOUR_HOUR_MS`; при `SIGNAL_TF="4h"` = 14_400_000 = байт-в-байт; (б) ПОСЛЕ
  `run_4h_cycle` звать авто-warm для монет вселенной без активного сетапа.
- **Горн** — новый durable-интент `WARM_AUTO_NOW` в `config_log` (single-shot + ack, зеркало `WARM_APPLY`);
  адаптер кладёт его по команде; движок консюмит рядом с `maybe_warm` (каждый тик) → тот же авто-warm.
  **Кнопку переиспользуем:** расширить адаптер `bot.py:_scan_now` (команда `scan_now` уже есть в ядре+консоли)
  — после скаут-скана дополнительно класть `WARM_AUTO_NOW`. Новую команду ядра/консоли НЕ заводим.
- **Общее ядро** — вынести per-coin `auto_eligible`-подхват (петля `_warm_one`) в вызываемую функцию, звать
  из 3 мест: bootstrap (гейт `WARM_ON_START`, как сейчас) / граница SIGNAL_TF (самоход) / интент (горн). Гейты
  `_warm_gate` (пауза/kill/equity) + `_warm_cap_ok` + `has_active` — переиспользуются, НЕ дублируем.

## Аудит плана (5 линз, адверсариально) — ветка А
- **Архитектура:** добавляем КАДЕНЦИЮ существующего безопасного инварианта (`warm_start_auto`), не новую
  торговую логику. Порядок на границе: `run_4h_cycle` (свежий скан/постановка) ПЕРВЫМ → warm ВТОРЫМ; монету,
  что цикл поставил, warm видит `has_active`→skip (нет двойной постановки). warm подберёт ровно то, что
  no-backfill курсор пропустил (сформировалось до входа) — цель.
- **Швы:** адаптер↔движок только интентом (шов чист). Новый `WARM_AUTO_NOW` — single-shot+ack (иначе
  повторные постановки): моделируем по `WARM_APPLY`.
- **Баги/отказы:** `_warm_gate`/cap/`has_active`/write-ahead `_commit_place` сохранены; сеть per-coin
  fail-soft; сбой warm на границе — try/except, не валит движок (как `run_4h_cycle`).
- **🔴 Ведение открытых позиций (урок 18.07):** warm ставит ТОЛЬКО для монет БЕЗ активного сетапа
  (`has_active`→skip) — открытые позиции/ведение не трогает. Побочно: больше поставленных сетапов = больше
  экспозиции при будущем редеплое — смягчено пином/гейтом (Веха 2) + Postgres-персистентностью (эта сессия).
- **Флот/законы/секреты:** гейт-флаг OFF по умолчанию + граница-тождество при 4h → Персиваль/Галахад
  **байт-в-байт** (страж-дрейфа + тест это ГАРАНТИРУЮТ — критическая проверка). Секретов нет; аудит — прежние
  события. Портал клиента горна не видит (Оператор-only через существующий RBAC scan_now).

## Риски / границы
- SIGNAL_TF-параметризация трогает ТОЛЬКО ритм warm-каденции. Полный ре-тул движка на произвольный ТФ
  (`fetch_4h_series`, `is_4h_close`, курсоры) — НЕ в этой задаче (SIGNAL_TF=4h; закладываем параметр, не
  переписываем движок).
- Геном-правка у общего кода флота: страж-дрейфа + байт-в-байт тест обязательны (иначе риск Персивалю).
