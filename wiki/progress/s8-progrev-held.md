# Порция №2: прогрев held — ПЛАН (подход подписан Куратором в КАНОНЕ ОЧЕРЕДИ 2026-07-23)

Куратор в каноне описал подход и класс проблемы (смягчённый родственник 18.07). «Обычный цикл:
план-строки → код → тесты vs вендор → деплой Борса → сверка». Полный персист scout.db/coins.json —
техдолг-гейт до реальных денег, НЕ здесь.

## Проблема (доказана вживую при редеплое баров)
На полном редеплое Борса scout.db эфемерна → сбрасывается. В engine-режиме `tick`
(`dynamic_universe.py:121-124`) гейтит запись дешёвым курсором: пока `scan_ms == 0` (скаут ещё не
сканил), провайдер **возвращается БЕЗ записи coins.json**. Итог: ~20 мин (04:18→04:36 в живом
редеплое) движок на дефолтных 16 монетах, **held-позиции НЕ в рабочей вселенной → движок их не ведёт**
(трейлинг/таймаут не работают; защита только биржевыми стопами). Postgres спас карточки позиций
(reconcile 4/4 kept), потому терпимо на демо — но окно неведения реальное.

## Инсайт (Куратор)
held-позиции известны **с первой секунды** из `build_monitor` (читает БД движка / Postgres), НЕЗАВИСИМО
от скаута. `tick(now_mono, held)` уже получает `held` (bot.py: `mapper.held_symbols(monitor)`). ⇒
провайдер может записать вселенную из held СРАЗУ, не дожидаясь скана.

## Дизайн (0-vendor, только адаптер)
В `tick`, ДО скан-гейта: если есть held И ещё ничего не писали (`self._written_sig` пуст) →
`_maybe_write(now_mono)` немедленно. `_maybe_write` уже берёт `symbols = стек ∪ held` → запишет held.
Своих scan_list-mb у held на старте нет → `coin_block` фолбэкнет на дефолт-бары (валидны) — движок
ВЕДЁТ позиции с 1-й минуты; per-coin бары приедут, когда скаут отсканит (обычный путь перезапишет).
- **Одноразово:** как только реальный набор записан (`_written_sig` не пуст) — прогрев не вмешивается,
  дальше строго обычный путь по скану (не ломает анти-thrash, не дёргает лишний рестарт в стабильном ходу).
- **Не двигаем `_last_scan_ms`:** скан-гейт по-прежнему сработает на первый реальный скан → per-coin
  вселенная перезапишет прогрев (min_write_s throttle сгладит соседние записи).
- **Источник-агностично:** тот же гейт у scout-пути; прогрев из held полезен обоим (Борс — первый).

## Тесты (vs НАСТОЯЩИЙ вендор + юнит провайдера)
- boot: held непуст + `scan_ms == 0` → coins.json содержит held СРАЗУ (дефолт-бары), БЕЗ скана.
- held пуст ИЛИ `_written_sig` уже задан → прогрев НЕ пишет (нет спурьёз-записи).
- после прогрева: скаут даёт новый `scan_ms` → обычный путь перезаписывает per-coin вселенной (held с
  барами, если placeable).
- held в coins.json проходит `config.validate` (дефолт-блок валиден).
- регресс: в устоявшемся ходу (набор записан) поведение байт-в-байт прежнее.

## Инварианты
- 0-vendor (только вызовы; страж-дрейфа = 2 дельты). Упрёмся в вендор → СТОП.
- Персиваль/Галахад не трогаем (у них провайдера нет). 🔒 Демо.
- Полный персист (том/прогрев из ядра) — техдолг-гейт до реальных денег, НЕ в этой порции.

## Порядок
план → код (`tick` прогрев + helper при нужде) → тесты vs вендор → ruff/страж-дрейфа → merge →
деплой Борса → живая сверка: редеплой Борса → held в coins.json с первой минуты (окно 20мин→~тик),
позиции ведутся сразу; стражи неизменны.
